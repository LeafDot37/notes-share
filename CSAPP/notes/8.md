# 进程

![1779629607278](images/8/1779629607278.png)

![1779691448368](images/8/1779691448368.png)

![1779691591759](images/8/1779691591759.png)

具体来说，属于进程上下文的有寄存器，进程表，文件表等

我们可以将进程上下文分为三大类：用户级上下文、寄存器上下文和系统级上下文。：

用户级上下文是进程作为独立程序运行时所依赖的数据和代码环境。

正文段：即代码段，包含 CPU 要执行的机器指令。
用户数据段：包含全局变量、静态变量等。
用户堆栈： 用于函数调用、局部变量存储。
共享内存段：如果进程使用了共享内存通信，这部分映射也属于上下文。

寄存器上下文是最核心的“现场”信息。当发生进程切换（Context Switch）或中断时，CPU 必须保存这些值，以便下次恢复执行时能接续上。

程序计数器：指向下一条要执行的指令地址。这是恢复执行的关键。
栈指针：指向当前栈顶，决定了当前函数的栈帧位置。
通用寄存器： 这些寄存器存放着正在计算的中间结果。
状态寄存器：包含进位标志、零标志、中断允许位等。它决定了条件跳转（if/else）的结果。
段寄存器 / 基址寄存器：用于内存寻址，确定代码和数据在内存中的位置。
浮点/SIMD 寄存器：如果进程进行了浮点运算或向量计算，这些状态也必须保存。

系统级上下文

这是内核为了管理该进程而在内核空间中维护的数据结构。这部分对用户不可见，但对操作系统至关重要。

进程控制块

这是进程存在的唯一标志，包含了进程的所有元数据：
进程标识符： 唯一的 ID。
进程状态：运行、就绪、阻塞、僵尸等。
优先级与调度信息：决定谁先获得 CPU。
记账信息：CPU 使用时间、创建时间等。
信号屏蔽字与待处理信号：记录哪些信号被阻塞，哪些信号到了但还没处理。

内存管理上下文

页表：虚拟地址到物理地址的映射表。切换进程时，必须切换页表基址寄存器。
内存描述符：记录进程的代码段、数据段、堆、栈的起始和结束地址（VMA 链表）。
打开文件表：这是一个数组，索引是文件描述符，内容是指向内核文件对象的指针。它记录了进程打开了哪些文件、读写偏移量在哪里。
文件系统上下文：当前工作目录、根目录的 inode 指针。

硬件与内核栈

内核栈：每个进程在内核态执行时（如系统调用、中断处理）使用的栈。用户栈不能用于内核操作，因为不安全且大小受限。

线程本地存储指针：指向线程私有数据的指针。

### fork例子

![1779694503007](images/8/1779694503007.png)

```c
#include <sys/types.h>
#include <unistd.h>
#include <stdio.h>   // 引入 printf 所需的头文件
#include <stdlib.h>  // 引入 exit 所需的头文件

int main() {
    pid_t pid;
    int x = 1;
  
    pid = fork();
  
    if (pid == 0) {  /* Child 子进程 */
        printf("child : x=%d\n", ++x);
        exit(0);
    }
  
    /* Parent 父进程 */
    printf("parent: x=%d\n", --x);
    exit(0);
}
```

结果如下

```
leafd_dot@Leaves:/mnt/d/DOC/CSAPP/test$ gcc temp.c -o temp
leafd_dot@Leaves:/mnt/d/DOC/CSAPP/test$ ./temp
parent : x=0
child: x=2
```

简单来说，fork会将未执行fork的父进程完全拷贝给子进程，由于fork函数在父进程中会创建子进程同时返回子进程pid，在子进程中不再创建子进程直接返回自身pid，于是在子进程中，`pid = fork()`的结果发生变化，进而有了下面的结果。

值得注意的是，系统将并发进行子进程和父进程，子进程和父进程输出顺序理论上是随机的，但是实际上，一般都是父进程先输出，保证减少CPU调度

### fork虚拟内存

![1779698203353](images/8/1779698203353.png)

![1779698241073](images/8/1779698241073.png)

左侧与右侧（虚拟地址）： 进程 1（父）和进程 2（子）都有自己独立的虚拟地址空间。起初，它们指向同一个物理地址（中间上方的浅蓝色块）。

触发点（写私有的指令）： 假设进程 2 试图修改这个内存块中的数据（写入操作）。

故障发生：由于该物理页被标记为“只读”，CPU 会拦截这次写入，抛出一个保护故障

内核介入（故障处理程序）：操作系统捕获到这个异常，发现是因为“写时复制”引起的。操作系统会在物理内存中分配一个新的页面（中间下方的黄色块）。将原来的数据复制到新页面中。更新进程 2 的页表条目，让它指向这个新的黄色页面，并将其权限改为“可写”。

最终结果：进程 1 继续使用原来的蓝色页面，进程 2 使用新的黄色页面。从此两者互不干扰，实现了数据的私有化。

## 回收子进程（wait）

![1779699428040](images/8/1779699428040.png)

![1779699873546](images/8/1779699873546.png)

在一个进程中断后，其上下文关系用户空间释放但是其内核空间仍然保留

用户空间释放：代码段、数据段、堆栈（用户态部分）确实被释放了，不再占用 CPU，也不再执行指令。

内核空间保留：在内核中的进程控制块仍然保留着

这个块里存着：进程 ID、退出状态码、资源使用统计信息等。这些信息就是留给父进程来“收尸”的。如果进程一死就彻底消失，父进程调用 `wait` 时就找不到它了，也就无法知道它是正常退出还是异常崩溃。

首先顺序创建进程，将pid存入数组，接着倒序等待回收，在回收的循环中，只有确定了当前子进程正常回收了，才会继续去回收下一个

![1779700739357](images/8/1779700739357.png)

WNOHANG：非阻塞轮询（W no hang）

作用： 如果子进程结束，则返回pid，如果没结束，则返回0并继续进行

适用于父进程需要同时处理多项任务的情况。比如一个服务器程序，使用这个选项，父进程可以在主循环中不断“偷窥”子进程的状态，而不会被卡死。

WUNTRACED：捕获停止状态（W untraced）

作用： 默认情况下，`waitpid` 只关心子进程是否结束，加上这个选项后，子进程受到信号（`SIGSTOP` 或 `SIGTSTP`用户按了`ctrl+z`）暂停时也会返回pid

主要用于实现调试器（Debugger）。当你在 GDB 里打断点时，程序其实是被暂停了，调试器就是通过这个机制知道“哦，目标进程停下来了，我现在可以检查它的内存了”。

WCONTINUED：捕获恢复运行（W continued）

作用：如果一个子进程之前被暂停了，收到 `SIGCONT` 信号准备继续运行时，`waitpid` 会捕捉到这个事件并返回。

同样多用于调试器或作业控制 Shell，用来追踪进程状态的完整生命周期（运行 -> 暂停 -> 继续 -> 结束）。

WNOHANG | WUNTRACED 组合使用

不阻塞的条件下，如果有子进程终止或者暂停则返回pid，即如果集合里的子进程既没有死也没有停，函数立即返回 0；一旦有任何一个子进程发生状态变化（无论是死是停），函数都会立即返回该进程pid

## 加载并允许程序（execve）

![1779701775893](images/8/1779701775893.png)

![1779705097089](images/8/1779705097089.png)

在这个栈结构中，最上方两个存的是环境变量和命令行的真实数据，下边是数组，指向真实数据

即argv变量是个二级数组，其存的是一系列指向真实数据的地址，通过进一步索引得到实际的数据

![1779730165156](images/8/1779730165156.png)

当 `execve` 执行时，内核中的加载器（Loader）会做以下清理工作：

代码段 (.text)： 原来程序的机器指令被丢弃。

数据段 (.data/.bss)： 原来的全局变量、静态变量被清空。

堆 (Heap) 和 栈 (Stack)： 原来程序动态分配的内存、函数调用的栈帧全部被销毁，重新初始化。

虽然程序换了，但操作系统认为这还是同一个进程，所以以下关键属性会被完整保留：

PID (进程ID)：这是最重要的。如果你调用 `execve` 前 PID 是 100，调用后（变成了 `/bin/ls`）PID 依然是 100。

父进程关系： 它的爸爸还是原来的那个爸爸。

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/types.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork(); // 1. 创建一个子进程

    if (pid < 0) {
        perror("Fork failed");
        return 1;
    }

    if (pid == 0) {
        // --- 这里是子进程 ---
        printf("[子进程] 我是原来的程序，PID=%d\n", getpid());

        // 准备参数
        // argv[0] 通常是程序名，argv[1...] 是参数，最后必须是 NULL
        char *argv[] = {"/bin/ls", "-l", "/tmp", NULL};

        // 准备环境变量 (这里简化了，实际通常继承父进程 environ)
        char *envp[] = {"PATH=/usr/bin:/bin", NULL};

        printf("[子进程] 即将变身...\n");

        // 2. 关键步骤：调用 execve
        // 如果成功，这行代码之后的所有代码都不会被执行！
        // 当前进程的代码段、数据段会被 /bin/ls 覆盖
        int ret = execve("/bin/ls", argv, envp);

        // 3. 只有 execve 失败时，才会执行到这里
        perror("Execve failed");
        return 1;
    } else {
        // --- 这里是父进程 ---
        printf("[父进程] 我创建了子进程(PID=%d)，我在等它结束...\n", pid);
        wait(NULL); // 等待子进程结束
        printf("[父进程] 子进程已结束。\n");
    }

    return 0;
}
```

利用execve可以将子进程变成需要的代码，同时不影响主进程使用，回收也非常方便。注意execve如果成功则不返回值，如果失败则返回-1，可以接收返回值进行调试

pending和blocked都是对于接收信号的进程来说的

对于一个信号，按照其是否存在于blocked中将其区分为是否被拦截

未被拦截：信号进入 `pending` 集合，内核发现设防，于是立刻把它拿出来处理（递达），处理完马上从 `pending` 里清除。这个过程极快，甚至感觉不到它在 `pending` 里待过。

被拦截：信号进入 `pending` 集合，内核看到这个信号被 Block 了，于是保留它在 `pending` 里的标记，暂时不去动它。它会一直待在那儿，直到解除阻塞。

主要对于A接收到信号后的进程进行分析

第一次切换（进入 Handler）：当信号到达时，CPU 从用户态（User Mode）陷入内核态（Kernel Mode）。内核负责保存 `main` 的寄存器状态，修改程序计数器指向 `handler`，然后返回用户态。这就像一次微型的“进程调度”。

第二次切换（退出 Handler）：当 `handler` 执行完毕，CPU 再次陷入内核态。内核负责恢复之前保存的 `main` 的寄存器状态，把程序计数器指回 `main` 被打断的地方），然后再次返回用户态。

信号处理程序的运行机制，在底层原理上更接近于进程切换，而不是简单的函数调用。

# 非本地跳转

![1779775037666](images/8/1779775037666.png)

![1779775096773](images/8/1779775096773.png)

setjmp，存档

第一次直接调用时：它像一个普通函数一样工作，返回0。这表示“存档成功”。
被 `longjmp` 唤醒时：通过`longjmp` 跳回到了这里，它会再次“返回”，但这次返回的值是 `longjmp` 传过来的那个值（非零）

longjmp传送

它从缓冲区`j` 中取出之前保存的寄存器值（PC, SP, BP 等）。
它强制修改 CPU 的寄存器，让程序计数器（PC）指回当初`setjmp` 所在的位置。
它将参数 `i` 放入返回值寄存器（如 x86 的 `%eax`），作为 `setjmp` 的返回值。
程序看起来就像是刚刚执行完 `setjmp` 并返回了数值 `i`。

因此初次调用setjmp和之后每次调用longjmp都会使setjmp返回，是调用一次返回多次
而longjmp被调用后，函数中就将跳转到setjmp，因此不会返回
