## 符号解析

### 整体过程

![1779179646584](images/7/1779179646584.png)

符号分为定义符号和引用符号，这里只包括了全局变量和函数，符号定义存放在符号表中，符号解析就是关联符号引用和符号定义。

这个阶段的核心任务就是汇总。链接器会扫描所有输入的目标文件（`.o`）和库文件，将它们各自内部的符号表提取出来，合并成一张全局符号表。

在汇总的过程中，链接器主要干两件事：

1. 汇总符号：把所有文件里的符号定义（函数名、全局变量等）收集到一起。
2. 解析绑定：拿着这张汇总后的总表，去核对每一个“符号引用”（比如你在代码里调用了 `printf`）能不能找到对应的“定义”（`printf` 到底在哪）。如果找不到，就会报 `undefined reference` 错误；如果同一个符号被重复定义，就会报 `multiple definition` 错误。

### 符号分类

![1779289912315](images/7/1779289912315.png)

使用静态属性定义的全局变量和函数是本地符号，无法被其他程序调用，可以防止命名冲突

场景四：

* 链接器会丢弃模块 2 中的弱符号定义，所有对 `x` **的引用都指向模块 1 中的那个** `int`。

## 重定位

### 整体过程

![1779180684342](images/7/1779180684342.png)

重定位是在符号解析完全结束之后才开始的。此时，全局符号表已经汇总完毕，且所有的符号引用都已经找到了归宿。

重定位阶段主要干两件事：

1. 分配运行时地址（布局）：链接器根据汇总后的信息，决定每个代码段（`.text`）、数据段（`.data`）在最终可执行文件内存中的绝对地址。比如，它确定 `main` 函数最终会被加载到内存的 `0x401000` 处。
2. 修改代码中的地址（填空）：编译器在生成 `.o` 文件时，遇到外部调用会先留一个“占位符”（比如 `call 0x00000000`）。重定位就是把这些占位符，替换成上一步计算出来的真实内存地址（比如把 `0x00000000` 改成 `0x401000`）。

![1779296303655](images/7/1779296303655.png)

![1779296343273](images/7/1779296343273.png)

![1779296491349](images/7/1779296491349.png)

![1779296636043](images/7/1779296636043.png)

a: R\_X86\_64\_32 array # 可重定位条目
f: R\_X86\_64\_PC32 sum-0x4 # 可重定位条目

并不是真的写在代码中，而是被存储在了.rel.data,.rel.text节当中。

r.offset = 0xf
r.symbol = sum
r.type = R_X86_64_PC32
r.addend = -4

分别表示

需要修改的位置0xf即e8后的00 00 00 00占位地址开始部分

目标符号的名字sum即这里应该填入符号sum对应的地址

重定位类型R_X86_64_PC32即填入相对地址，从当前指令的下一条指令开始，到 `sum` 函数入口的距离（偏移量）

加数-4即调整当前位置0xf到下一条指令的位置即0x13

整体就是在说：在文件偏移 `0xf` 处，有一个对 `sum` 的调用。因为使用的是相对寻址（PC32），请你算出 `sum` 距离下一条指令（当前地址+4）有多远，把这个距离差值填进去。

### 具体过程

![1779297571042](images/7/1779297571042.png)

即如果是相对引用，填入的值=目标地址+补偿值−当前指令的下一条指令地址

如果是绝对地址，填入的值=目标变量的绝对地址+补偿值

为什么要取下一条指令为基准呢，这是由于之前的流水线机制，在取指阶段PC就指向了下一个该被执行的指令

![1779297908209](images/7/1779297908209.png)

![1779297916008](images/7/1779297916008.png)

![1779297931001](images/7/1779297931001.png)

相对地址是为了对应于call指令的底层。在 x86-64 的 `call` 指令中，硬件计算跳转目标的公式是：`目标地址 = 下一条指令地址 + 填入的立即数`

在相对地址中，加数一般是为了抵消指令的长度，在绝对寻址中，加数一般是为了取变量内部某成员

在命令行直接运行 `./main`，系统默认只去标准路径（如 `/lib`, `/usr/lib`）找库，环境变量 `LD_LIBRARY_PATH`与在哪找库有关

静态库意味着将库代码写入了目标程序，而动态库并没有直接写入，在加载或者运行时才写入或者说使用

动态库.so由多个.c文件生成，在主程序中只是使用其符号而并没有直接调用，在加载使用的时候才动态调用，因此更新动态库后，只要接口（输入输出类型）不变，就不需要将使用该库的程序即主程序重新编译

## 编译时打桩

![1779345280906](images/7/1779345280906.png)

![1779345286923](images/7/1779345286923.png)

编译时打桩就是利用 C 语言的预处理机制（宏定义），在代码编译之前，把所有对标准函数（如 `malloc`）的调用，悄悄替换成你自己写的函数（如 `mymalloc`），进而使用自己写的函数中的调试信息进行调试

## 链接时打桩

![1779345747298](images/7/1779345747298.png)

![1779345753259](images/7/1779345753259.png)

链接时打桩利用了链接器的 `--wrap` 选项，该选项会告诉链接器：

* 所有 `malloc` 的引用，都重定向到 `__wrap_malloc`
* 所有 `__real_malloc` 的引用，都重定向到真正的 `malloc`
* 写的 `malloc()`->链接器变成`__wrap_malloc()`->你定义的函数
* 写的 `__real_malloc()`->链接器变成 `malloc()` ->系统库的真实函数

## 加载/运行时打桩

![1779346352101](images/7/1779346352101.png)

![1779346358549](images/7/1779346358549.png)

环境变量进行指定共享库的优先级

在代码中

    // 使用 dlsym 获取真正的 free 地址
    // RTLD_NEXT 是关键：它告诉动态链接器跳过当前的库，
